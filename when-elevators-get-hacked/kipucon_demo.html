System architecture

Three-layer OT stack: SCADA dispatcher → PLC controllers → physical I/O. All connected via unauthenticated Modbus/TCP.

PLC state machine

Each elevator runs an independent 100ms scan loop. The attack bypasses the state machine entirely via a direct register write.

IDLE
Waiting. Dequeues one floor command per scan cycle.
MOVING_UP
Advances floor+1 every 0.6s until target reached.
MOVING_DOWN
Advances floor-1 every 0.6s until target reached.
DOOR_OPEN
Holds doors open 1.2s. HR2=1.
DOOR_CLOSING
0.4s transition. HR2=2 → 0, back to IDLE.
FAULT
Auto-recovery after 3s. Clears HR4.
LOCKED_FAULT ← attack outcome
Terminal state. Set by direct HR4 write (Modbus FC=6). No auto-recovery path. Physical technician reset required.
DOOR_CLOSING → IDLE
Normal loop-back. Dequeues next cmd.
Normal loop: IDLE → MOVING_UP/DOWN → IDLE → DOOR_OPEN → DOOR_CLOSING → IDLE Attack path: IDLE → LOCKED_FAULT (HR4 write — skips state machine entirely)
Why the queue matters
The buffer is capped at 8 slots — realistic for embedded PLC hardware. SCADA writes to the queue; the PLC drains it one command per IDLE cycle. Buffer overflow drops commands silently — no error propagates back to SCADA. An attacker can saturate the queue so legitimate floor calls are never processed — the elevator becomes unresponsive without triggering a visible fault.

Attack flow — 3 phases, one target

ELV-1 is taken offline. ELV-2 and ELV-3 remain operational — a realistic partial-compromise scenario.

PHASE 1 — Passive recon
Sniff Modbus/TCP poll traffic
Attacker sits on the flat OT network. SCADA polls HR0–HR7 from every PLC every 100ms — unauthenticated, unencrypted. Attacker reads current floor, queue depth, fault register address, and cycle counter from passive observation alone.
Sniffed Modbus/TCP poll → ELV-1 HR0=current_floor:1 HR1=target:1 HR4=fault_code:0x00 HR5=queue:0 No authentication required.
PHASE 2 — Command buffer flood
Saturate the PLC cmd_queue (DoS)
Attacker sends rapid Modbus FC=15 (Write Multiple Registers) commands with random floor values. The 8-slot buffer fills. Legitimate SCADA floor calls are now dropped. The elevator starts bouncing to attacker-supplied floors — erratic but not yet faulted.
[ATTACKER] WRITE FC=15 → floor=07 [ACCEPTED] [ATTACKER] WRITE FC=15 → floor=03 [ACCEPTED] ... [ELV-1] CMD BUFFER OVERFLOW — command dropped [ATTACKER] WRITE FC=15 → floor=09 [DROPPED] [SCADA] CALL floor 5 → ELV-1 [REJECTED] [SCADA] CALL floor 5 → ELV-2 [OK] ← fallback
PHASE 3 — Register write exploit
Write fault_code=0x10 to HR4
A single Modbus Write Single Register packet (FC=6) targets HR4. No authentication. No input validation. fault_code 0x10 = critical controller error. The PLC transitions to LOCKED_FAULT immediately — bypassing the entire state machine.
WRITE FC=6 addr=HR4 value=0x10 → ELV-1 PLC Result: state = LOCKED_FAULT (no auto-recovery) [ATTACK] ELV-1 FAULT REGISTER WRITTEN fault_code=0x10 | Controller LOCKED
IMPACT
ELV-1 locked — passengers trapped — manual reset required
ELV-1 frozen between floors. Doors closed. SCADA HMI shows LOCKED_FAULT / 0x10. No auto-recovery. Only a technician at the physical controller cabinet can clear the fault. ELV-2 and ELV-3 absorb all traffic — building degraded but not halted.

Defense — how to stop this

None of these mitigations require replacing the PLCs. They are configuration and network changes.

01
Network segmentation
Place elevator PLCs on a dedicated VLAN. Firewall rule: SCADA server → PLC IPs on port 502 only. Attacker on the corporate network cannot reach the OT segment at all.
02
Modbus firewall — Function Code allowlist
Deploy an OT-aware firewall (Claroty, Dragos, Nozomi) that inspects Modbus packets. Allow FC=3 (read) from SCADA IP only. Block FC=6 (write register) from any unauthorized source.
03
Write-protect HR4 at PLC config level
Siemens S7-1500 and Allen-Bradley ControlLogix support register-level access control in engineering config. HR4 (fault_code) should be read-only from the network — writable only via local engineering tool with password.
04
Queue rate limiting + anomaly detection
PLC-side logic: if cmd_queue fills >6 slots within 2 seconds, raise SCADA alarm. Real-world average: 1–3 floor commands per minute. 24 commands per second is an obvious anomaly — detectable without ML.
The root cause (Modbus RFC 6833)
Modbus/TCP was designed in 1979 for serial communication in isolated factory floors — adversarial networks were not a design consideration. RFC 6833 explicitly acknowledges this. The protocol has no session concept, no authentication mechanism, and no packet signing. Any device on the same network that can reach TCP port 502 can read and write any register on any PLC with no credentials. This is not a bug — it is the intended behaviour of the protocol. Security must be enforced by the network, not the protocol.
